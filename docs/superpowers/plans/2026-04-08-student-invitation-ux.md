# 학생 초대 UX 개선 구현 계획

## 목표

교사가 학생 초대를 더 빠르게 처리할 수 있도록 학생 관리 화면을 초대 중심 허브로 재구성한다.

- **목표 1:** 교사가 30초 안에 학생 1명을 초대할 수 있다.
- **목표 2:** 교사가 2분 안에 여러 명을 일괄 초대할 수 있다.
- **목표 3:** 초대 상태 확인, 재전송, 만료 처리를 한 화면에서 할 수 있다.

이번 작업은 **기존 백엔드 계약을 최대한 유지**하고, 가능한 한 **프론트엔드 강화 중심**으로 구현한다.

### 현재 부족한 것

- 학생 목록 응답인 `GET /users/students`는 초대 상태, 만료일, 재전송 횟수를 내려주지 않는다.
- 학생 목록 테이블은 초대 상태 컬럼과 빠른 액션이 없다.
- 현재 단건 초대 폼은 성공 액션이 `링크 복사` 수준에 머물러 있고, `QR 보기`, `공유용 텍스트 복사`가 없다.
- 대량 초대는 파일 업로드 후 서버 결과만 보는 형태라서, 사용자가 붙여넣기 → 검증 → 유효 행만 생성 흐름을 통제할 수 없다.
- 재전송/강제 만료를 위한 정리된 API가 없다.
- 가입 화면은 초대 대상 표시와 자동 로그인은 되지만, 비밀번호 규칙의 실시간 피드백이 약하다.

### 구조상 중요한 관찰

- `StudentList`는 각 행에서 `useStudent`, `useGrades`, `useAttendanceToday`, `useLatestSpecialNote`를 추가 호출한다.
- 여기에 초대 상태까지 행 단위 개별 조회로 추가하면 N+1 문제가 더 심해질 가능성이 높다.
- 따라서 **초대 상태는 목록 API에 함께 실어 보내거나, 클래스 단위 상태 조회 API를 별도로 두는 방식**이 적합하다.

## 선결 결정 사항

### 결정 1 — “필수 입력 4개”와 기존 API 계약의 충돌

사용자 목표는 단건 초대의 필수 입력을 `이름, 학년, 반, 번호`로 제한하지만, 현재 `POST /users/students` 계약과 PRD는 `email`을 필수로 요구한다.

- 현재 PRD 기준: 학생 계정은 **실제 이메일 기반 초대**로 생성한다.
- 현재 API 기준: `StudentCreate`에 `email`이 필수다.

즉, 아래 둘을 동시에 그대로 만족시키기는 어렵다.

1. `POST /users/students`를 그대로 유지한다.
2. 필수 입력을 `이름, 학년, 반, 번호`만으로 제한한다.

### 권장 해석

이번 구현 계획은 다음 우선순위를 권장한다.

- **권장안 A:** 백엔드 계약은 유지하고, 실제 구현 시 필수 입력을 `이름, 학년, 반, 번호, 이메일`로 둔다.
- **권장안 B:** 정말 4개만 필수로 하려면, 별도 학생 식별자/학교 alias/연락처 후입력 정책이 필요하며 이는 소규모 백엔드 보강 범위를 넘을 수 있다.

이 문서는 **권장안 A 기준**으로 작성하되, UI는 최대한 30초 목표에 맞게 입력 부담을 줄이는 방향으로 설계한다.

## 구현 전략

### 핵심 전략 1 — 학생 관리 화면을 초대 허브로 유지

새로운 별도 페이지를 만들기보다 `StudentListPage`를 중심으로 다음 3가지를 통합한다.

1. 단건 초대
2. 대량 초대
3. 초대 상태 관리

### 핵심 전략 2 — 대량 초대는 프론트 오케스트레이션으로 구현

대량 초대는 기존 `POST /users/students`를 반복 호출하는 방식으로 구현한다.

이 방식의 장점:

- 기존 생성 API를 그대로 활용할 수 있다.
- 행 단위 성공/실패/중복 상태를 프론트에서 즉시 보여줄 수 있다.
- 생성 직후 반환되는 `invite_url`을 모아 `초대 링크 일괄 복사`를 쉽게 제공할 수 있다.
- CSV 업로드, XLSX 업로드, 엑셀 표 붙여넣기를 모두 하나의 로컬 검증 파이프라인으로 묶을 수 있다.

### 핵심 전략 3 — 초대 상태는 “최신 초대 + 초대 이력 수”로 파생 계산

`user_invitations` 테이블에 새 필드를 추가하지 않고도 대부분의 요구를 만족할 수 있다.

- **상태:** 최신 초대의 `accepted_at`, `revoked_at`, `expires_at`로 계산
- **만료일:** 최신 초대의 `expires_at`
- **재전송 횟수:** 해당 사용자의 초대 레코드 개수 기반 파생값
- **최근 전송 시각:** 최신 초대의 `created_at`

이 접근을 쓰면 **DB 마이그레이션 없이**도 상태 조회/재전송 UX를 구현할 수 있다.

## 단계별 구현 계획

## 1) 백엔드 초대 상태/재전송 최소 보강

### 목표

프론트가 한 화면에서 초대 상태를 보여주고 재전송/만료 처리할 수 있도록, 목록 응답과 액션 API를 최소 범위로 확장한다.

- [x] `GET /users/students` 응답에 초대 요약 필드를 추가한다.
- [ ] 추가 필드 후보:
  - `email`
  - `account_status`
  - `invite_status` (`pending` | `accepted` | `expired`)
  - `invite_expires_at`
  - `invite_sent_at`
  - `invite_resend_count`
- [x] 학생 목록 조회 시 사용자별 최신 초대 1건과 전체 초대 수를 함께 계산하는 서비스 쿼리를 추가한다.
- [x] `POST /users/students/{student_id}/invitation/resend`를 추가한다.
- [x] 재전송 시 기존 활성 초대는 `revoked_at`으로 종료하고 새 초대를 발급한다.
- [x] 응답에는 새 `invite_url`, `invite_expires_at`, `invite_resend_count`를 포함한다.
- [x] `POST /users/students/{student_id}/invitation/expire`를 추가한다.
- [x] 강제 만료는 최신 활성 초대에 `revoked_at`을 설정하고 프론트에서는 `만료`로 표시한다.
- [x] 모든 예외는 `AppException`과 `{ detail, code }` 계약만 사용한다.

### 구현 메모

- 별도 `resend_count` 컬럼을 추가하지 않고 초대 이력 개수 기반 파생값으로 계산한다.
- `accepted` 상태는 초대 레코드뿐 아니라 `user.is_active`와도 일관되게 맞춘다.
- 향후 학부모 재전송까지 확장할 가능성을 고려해 서비스 함수는 `user_id` 단위로 설계하고 라우터만 학생 기준으로 노출하는 편이 안전하다.

## 2) 단건 초대 UX 재구성

### 목표

교사가 학생 관리 화면에서 단건 초대를 빠르게 만들고, 생성 직후 공유 액션을 바로 수행하도록 한다.

- [x] `StudentListPage` 상단 CTA를 `학생 추가`에서 `학생 초대` 중심으로 재정렬한다.
- [x] 단건 초대 UI를 현재 모달에서 유지하거나 드로어로 바꾸되, 입력 흐름은 더 짧게 만든다.
- [x] 현재 선택된 학급을 기본값으로 사용해 `학년/반` 재선택을 최소화한다.
- [x] 단건 초대 필드는 기본적으로 아래 순서로 정리한다.
  - `이름`
  - `학생 이메일` 또는 결정된 대체 연락 수단
  - `번호`
  - 선택 정보: `생년월일`, `학부모 초대`
- [x] 성공 화면을 별도 액션 카드로 분리한다.
- [x] 성공 직후 액션 3개를 제공한다.
  - `링크 복사`
  - `QR 보기`
  - `카카오/문자 공유용 텍스트 복사`
- [x] 성공 후 학생 목록을 갱신하고 해당 학생 행에 `초대 대기` 배지를 노출한다.

### 구현 메모

- `링크 복사`는 기존 `invite_url`을 재사용한다.
- `공유용 텍스트`는 `학생명`, `학급`, `번호`, `가입 링크`, `안내 문구`를 조합하는 순수 프론트 유틸로 구현한다.
- `QR 보기`는 신규 의존성 없이 해결하는 것을 1순위로 검토한다. 필요 시 경량 SVG 렌더러 유틸을 내부 구현한다.
- 학부모 초대는 기본 접힘 섹션으로 내려서 단건 학생 초대의 속도를 해치지 않게 한다.

## 3) 대량 초대 UX 재구성

### 목표

파일 업로드 후 결과만 보는 기존 방식 대신, “붙여넣기/업로드 → 즉시 검증 → 유효 행만 생성 → 링크 일괄 복사” 흐름으로 바꾼다.

- [x] 기존 `ExcelUploadModal`을 대량 초대 전용 UI로 교체하거나 확장한다.
- [x] 대량 초대 UI 안에 최소 2개 탭을 둔다.
  - `CSV 업로드`
  - `엑셀 표 붙여넣기`
- [x] 가능하면 `XLSX 업로드`도 같은 모달 안에 함께 수용한다. 프론트에 이미 `xlsx` 의존성이 있으므로 새 의존성 없이 처리 가능하다.
- [x] 로컬 파서 유틸을 추가해 아래 입력원을 하나의 행 모델로 통합한다.
  - CSV 파일
  - XLSX 파일
  - 탭 구분 텍스트 붙여넣기
- [x] 행 단위 즉시 검증을 프론트에서 수행한다.
- [x] 검증 항목:
  - 필수값 누락
  - 번호 범위(`1~100`)
  - 동일 배치 내 중복 번호
  - 동일 배치 내 중복 이메일
  - 학급 매핑 실패
- [x] 검증 결과를 `성공 가능 / 실패 / 중복 의심` 상태로 행별 표시한다.
- [x] `유효한 학생만 생성` 버튼은 현재 유효 행만 대상으로 동작한다.
- [x] 생성 시 프론트에서 `POST /users/students`를 동시성 제한으로 반복 호출한다.
- [x] 각 행의 최종 상태를 `성공 / 실패 / 중복 / 생성 중`으로 즉시 갱신한다.
- [x] 생성 완료 후 성공한 행의 `invite_url`을 모아 `초대 링크 일괄 복사`를 제공한다.

### 구현 메모

- 이 흐름은 기존 `/import/students`보다 초대 UX 요구사항에 더 잘 맞는다.
- `/import/students`와 `/import/students/xlsx`는 기존 일반 데이터 import 용도로 유지하고, 초대 UX의 주 경로는 프론트 오케스트레이션으로 가져가는 것이 적합하다.
- 대량 초대에서 학년/반 컬럼이 제공되면 선택 학급과 무관하게 행별 class 매핑이 가능하지만, 1차 구현은 **현재 선택 학급 기본 적용 + 다른 학급 행은 오류 처리**로 단순화해도 된다.

## 4) 학생 목록 내 초대 상태 관리 화면 완성

### 목표

학생 목록 화면 자체가 초대 운영 화면이 되도록 컬럼, 필터, 빠른 액션을 추가한다.

- [x] `StudentList` 테이블에 `초대 상태` 컬럼을 추가한다.
- [x] 상태 배지는 최소 3종을 지원한다.
  - `대기`
  - `수락`
  - `만료`
- [x] 각 행에 빠른 액션을 추가한다.
  - `재전송`
  - `링크 복사`
  - `만료 처리`
- [x] 상단 필터를 추가한다.
  - `대기만 보기`
  - `7일 내 만료 예정`
- [x] 재전송 성공 시 해당 행의 만료일/재전송 횟수/링크를 즉시 갱신한다.
- [x] 강제 만료 성공 시 상태를 즉시 `만료`로 갱신한다.
- [x] 목록 조회 구조를 정리해 초대 상태 때문에 추가 N+1 호출이 생기지 않게 한다.

### 구현 메모

- 가능하면 `StudentSummary`를 초대 정보를 포함한 관리용 타입으로 확장하고, `StudentList`가 `useStudent`로 계정 상태를 따로 가져오지 않게 정리하는 편이 좋다.
- 현재 `StudentList`는 이미 행당 여러 API를 호출하고 있으므로, 이번 작업에서 최소한 초대 상태만큼은 목록 응답으로 합치는 것이 중요하다.

## 추천 구현 순서

1. **백엔드 초대 상태/재전송 최소 보강**
2. **학생 목록 타입/API 확장**
3. **단건 초대 성공 액션 강화**
4. **대량 초대 프론트 파서 + 부분 성공 UI**
5. **학생 목록 상태 컬럼/필터/빠른 액션**
6. **가입 화면 실시간 비밀번호 안내**
7. **회귀 테스트 + E2E 검증**

이 순서가 좋은 이유:

- 먼저 상태 데이터를 안정적으로 만들면 이후 UI 설계가 단순해진다.
- 단건 초대가 먼저 개선되면 목표 1을 가장 빨리 달성할 수 있다.
- 대량 초대는 로컬 파서와 상태 UI가 많아 범위가 크므로 2번째 묶음으로 두는 편이 안전하다.

## 완료 기준

- 학생 관리 상단에서 단건 초대를 시작할 수 있다.
- 단건 초대 성공 후 `링크 복사`, `QR 보기`, `공유용 텍스트 복사`가 제공된다.
- 대량 초대에서 CSV 업로드와 엑셀 표 붙여넣기를 지원한다.
- 대량 초대에서 행별 검증 결과를 즉시 보여주고, 유효 행만 생성할 수 있다.
- 생성 완료 후 성공한 링크를 한 번에 복사할 수 있다.
- 학생 목록 한 화면에서 `대기`, `수락`, `만료` 상태를 보고 `재전송`, `링크 복사`, `만료 처리`를 수행할 수 있다.
- 가입 화면에서 초대 대상과 비밀번호 규칙이 명확히 보인다.
- 관련 백엔드/프론트엔드/E2E 테스트가 추가된다.

## 리스크 및 메모

- **가장 큰 정책 리스크:** 학생 생성 시 `email` 필수 계약과 “필수 입력 4개” 요구의 충돌
- **성능 리스크:** 학생 목록의 기존 행별 다중 조회 구조에 초대 상태까지 얹으면 응답성이 악화될 수 있음
- **구현 리스크:** QR 표시를 무의존으로 구현할지, 별도 의존성을 허용할지 사전 합의가 있으면 작업이 빨라짐
- **권장 단순화:** 재전송 횟수는 DB 컬럼 추가 대신 초대 이력 파생값으로 처리
- **범위 통제:** 1차 릴리스는 학생 초대 중심으로 마무리하고, 학부모 초대 상태 관리 확장은 후속 단계로 분리 가능
